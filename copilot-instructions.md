# Copilot 指示テンプレート (C++ 用)

目的
- AI によるコード生成時の具体的ルールと期待値を明確にする。

基本ルール
- 言語: C++17
- コンパイラ: GCC 準拠を想定（GCC 拡張を使う場合は明記する）
- リソース管理: RAII を用いること（スマートポインタ、スコープガード等）。裸ポインタでの手動解放を避ける。
- エラーハンドリング: 実処理は自動で実装しない。エラー箇所には `// TODO: handle error` を残す。
- 命名規約:
  - 定数: 大文字＋アンダースコア（例: MAX_BUFFER_SIZE）
  - 変数/関数: スネークケース（例: connection_timeout_ms, process_packet()）
  - 型/クラス: PascalCase（例: MessageParser）
- コメント方針: 実装の「なぜ」を簡潔に記述する。処理手順の逐次説明（「ループして〜」等）は避ける。また、日本語でのコメントを推奨する。
- 出力単位: 小さな関数／クラス単位に限定する。既存モジュールの不用意な書き換えは行わない。
- ハードコーディング禁止: マジックナンバー（例: 100, 1024 など）は直接埋め込まず、定数（大文字＋アンダースコア）か設定で管理すること。

テンプレート（AI に渡す指示例）
- 「C++17、GCC 準拠で、RAII を使ったクラス `BufferManager` を実装してください。エラー箇所には `// TODO: handle error` を残してください。命名規約は定数は大文字＋アンダースコア、変数はスネークケース、型は PascalCase を使ってください。コメントには『なぜこの設計か』を短く書いてください。出力は単一ファイル内で 1 クラスに限定してください。」

良い / 悪い（簡潔比較）
- 良い:
  - Value Object で責務を分離する。
  - RAII で資源を管理する（例: `std::unique_ptr`, `std::lock_guard`）。
  - 早期リターンでネストを浅く保つ。
  - コメントは設計の背景・理由を説明する。
- 悪い:
  - グローバル状態に依存する実装。
  - 裸ポインタで手動 delete を行う実装。
  - 処理そのものを逐次説明する冗長なコメント。

注意事項
- 秘密情報（鍵、資格情報、シークレット等）は絶対に含めない。
- 既存のプロジェクトにスタイルガイドやルールがある場合は、それを優先する旨を明示すること。

原則と設計方針
- YAGNI（You Aren't Gonna Need It）: 将来の拡張を見越した過剰な汎用化を避け、現時点で必要な機能だけを実装する。プラットフォーム特化は合理的な選択肢となり得る。
- OCP（Open/Closed Principle）: 機能拡張はできるだけ既存コードの修正を避け、拡張で対応できるようインターフェース設計を心掛ける。

付記
- 出力は最小限に留め、必要なら補足説明を別ファイルで求める。

Kvaser (CANlib) 使用時の追加ルール
- 使用ライブラリ: Kvaser の CANlib (C/C++) を想定。プロジェクトでは SDK のラッパーを作成して使用することを推奨する。
- リソース管理: CAN チャネルやハンドルは RAII ラッパーで管理する（open -> configure -> bus on -> use -> bus off -> close のライフサイクルを明確にする）。
- 初期化と終了: ライブラリ初期化やチャネルオープンの失敗は実処理を書かず `// TODO: handle error` を置く。
- 推奨 API 操作シーケンス（例）: `canOpenChannel` -> `canSetBusParams` -> `canBusOn` -> `canWrite`/`canRead` -> `canBusOff` -> `canClose`。
- スレッド安全性: 受信は専用スレッドかコールバックで処理し、共有状態は `std::mutex` / `std::condition_variable` で保護する。
- タイムアウト/リトライ: 受信/送信のタイムアウトは明示し、リトライやバックオフの実装は設計上の注釈を残す（詳細処理は `// TODO: handle error`）。
- ビルド/リンク: Kvaser SDK に含まれるヘッダとライブラリにリンクする旨をコメントで明示する（例: SDK の提供するライブラリ名を参照）。
- テスト/モック: ハードウェア依存部分は抽象化してモック可能にする（インターフェースを小さく保つ）。
- セキュリティ: 絶対に鍵や資格情報をソースに埋め込まない。

テンプレート（Kvaser 向け指示例）
- 「C++17、GCC 準拠で、Kvaser CANlib を使う RAII ラッパークラス `KvaserChannel` を実装してください。チャネルの open/close を RAII で管理し、各 CANlib 呼び出しのエラー箇所に `// TODO: handle error` を残してください。命名規約は既定に従い、受信は専用スレッドで行い、共有状態に対する同期保護を使ってください。」
